How the documentation web site is created

Brimstone uses Sandcastle Help File Builder (SHFB) to build the documentation web site. Sandcastle parses all of the XML comments in the code files to produce an API reference, and parses hand-written documentation in a help file markup language called MAML to produce the other pages.

The documentation is stored in the Brimstone solution itself under the BrimstoneDocumentation project. To open this project, you must have SHFB installed. Download the installer and be sure to install all of the optional components - particularly the SHFB and MAML IntelliSense extensions for Visual Studio - as these will make your life much easier.

The documentation is built only when the Documentation build configuration is selected. This is the only build mode under which the Brimstone library build generates XML documentation, and the only build in which BrimstoneDocumentation is actually compiled. The finished documentation is generated in the /docs folder in the root of the Brimstone solution. The build configurations are organized this way because building the documentation is a time-consuming process.

While the API Reference portion of the documentation is generated from the XML comments in the code source files themselves, the remaining articles are written separately in MAML. The MAML content files can be found in the Content folder, images are stored in the media folder and the table of contents (left pane hierarchy) can be configured via the ContentLayout.content file. Each .aml file constitutes one topic, each with its own unique GUID which is used for link cross-referencing.

A crash course in Sandcastle and MAML follows. Refer to the Sandcastle MAML Guide for additional help with MAML tag syntax.

 Important

When you submit documentation, only include changes to ContentLayout.content, media and .aml files. Do not include the entire auto-generated /docs folder in your pull request as this creates a lot of commit noise. We will recompile the documentation web site when your PR is merged.

How to create a new topic

  1. Open ContentLayout.content, right-click on the topic you wish to create a new topic next to or underneath and select Add Sibling/Child Topic -> Add Topic From a Template...

  2. Select the template that most suits the article you are going to write. If you are not sure which to pick, choose Conceptual, which gives the most generic layout. If you are creating a topic which will just be a parent to other topics with an introductory paragraph, choose Orientation. If you are creating a topic which will be a parent to other topics with no content at all, choose Add Empty Container Topic from the menu instead.

     Tip

    The difference between empty topics and Orientation topics is that empty topics do not have a page associated with them - instead, when the user clicks on an empty topic, it will redirect them to the first child. An Orientation topic, on the other hand, has text and can contain links to the child (or any other) topics.

  3. Choose a topic filename in UpperCamelCase with a similar name to your intended topic title.

  4. Once you have created the topic file, enter a Title for the topic in the Topic Properties section of the content layout editor. This is the title that will actually be displayed in the documentation tree and at the top of the documentation page.

Standard topic file layouts

These are the basic layout templates we use in the Brimstone documentation. They may be adapted for other purposes as needed.

For orientation topics (topics that are the parent of a related group of child topics):

Copy
<?xml version="1.0" encoding="utf-8"?>
<topic id="[guid-of-topic]" revisionNumber="1">
  <developerOrientationDocument
    xmlns="http://ddue.schemas.microsoft.com/authoring/2003/5"
    xmlns:xlink="http://www.w3.org/1999/xlink">

    <summary>
      <para>Put a one line summary of your topic here.</para>
    </summary>
    <introduction>
      <para>Put one or more introductory paragraphs here introducing the content.</para>
      <para>You can use as many or as few paragraphs as you need.</para>
    </introduction>
    <inThisSection>
      <content>
        <!-- Put links to all of the child topics here, one per line as shown -->
        <para><link xlink:href="cfbf62f4-cb3b-4697-8f98-221c39cd3232" /></para>
        <para><link xlink:href="64f4468c-70df-4827-b028-66d547ce7952" /></para>
      </content>
    </inThisSection>
 </developerOrientationDocument>
</topic>

For general topics:

Copy
<?xml version="1.0" encoding="utf-8"?>
<topic id="[guid-of-topic]" revisionNumber="1">
  <developerConceptualDocument
    xmlns="http://ddue.schemas.microsoft.com/authoring/2003/5"
    xmlns:xlink="http://www.w3.org/1999/xlink">

    <summary>
      <para>A one-sentence summary of the article.</para>
    </summary>
    <introduction>
      <para>An optional longer description of the article if you wish.</para>
      <para>You can use as many or as few paragraphs as you need.</para>
      <!-- This will cause a list of links to be automatically generated for any <section>s below with an address attribute -->
      <autoOutline lead="In this section:"/>
    </introduction>

    <section address="firstsection">
      <title>Section 1 title</title>
      <content>
        <para>
          First paragraph.
        </para>
        <para>
          As many paragraphs as you like...
        </para>
        <para>
        <!-- If your section has sub-sections, you can use <autoOutline /> to generate a list of sub-section links here -->
      </content>
      <sections>
        <section>
          <title>Place sub-sections in a sections tag like this</legacyBold>
          <content>
            <para>Sub-section text. Sub-sections are not collapsible.</para>
          </content>
        </section>
        <section>
          <!-- As many sub-sections as you like -->
        </section>
      </sections>
    </section>

    <section address="secondsection">
      <title>Section 2 Title</title>
      <content>
         <!-- Repeat for as many sections as desired -->
      </content>
    </section>

    <!-- This tag must be included even if it is empty -->
    <relatedTopics>
      <!-- Put links to other topics and external resources (web pages) here -->
    </relatedTopics>
  </developerConceptualDocument>
</topic>
How to preview a topic

You do not need to re-compile the entire documentation to view your changes. Simply open the Topic Previewer via the View -> Other Windows -> Topic Previewer Window option. When you save your document, click the Refresh icon or press F5 in the Topic Previewer to see a preview of the document. Note that some formatting may be different and links to .NET elements and the API Reference will render in bold text rather than as a link.

If you wish to preview the entire documentation, build the solution (press F7) in Documentation build configuration. This may take several minutes to complete. Once the compilation is finished, click View Help File on the SHFB Toolbar, accessible via the View -> Toolbars -> Sandcastle Help File Builder menu option. This will allow you to view the documentation in a web browser.

 Tip

By default, compiling the documentation will start an IIS Express server hosting the documentation web site with the homepage at http://localhost:12345/SHFBOutput_1572097774/index.html.

How to move a topic in the documentation tree

  • To move a topic to another location at the same level in the tree: Click the topic in the ContentLayout.content editor window, then use the green arrows in the toolbar above to move the page up or down.

  • To move a topic to another level in the tree: Right-click the topic, select Cut, click on the topic you wish to be the new topic parent, right-click and choose Paste as Child.

How to insert a link to another topic

You do not need to work with topic GUIDs directly. To insert a link to another topic page:

  1. Open the Entity References window via the View -> Other Windows -> Entity References Window menu option.

  2. Choose Table of Contents from the drop-down box to show the documentation tree.

  3. Drag the node of the topic you wish to include into your MAML document. The link tag will be inserted at the cursor position.

How to insert a link to a .NET element or an item in the API Reference

Code elements are referred to using a special syntax explained here. These can include links to external documentation such as the MSDN web site as well as the Brimstone API Reference. Entering these references by hand can be awkward, but you can insert them in a similar way to topic links:

  1. Open the Entity References window via the View -> Other Windows -> Entity References Window menu option.

  2. Choose Code Entities from the drop-down box to show the topic tree.

  3. Search for the code element you wish to refer to in the Find box.

     Note

    Only elements with XML documentation will be listed in the Entity References window. To link to elements without XML documentation, you must enter the refrence syntax by hand.

  4. Drag the node of the element you wish to link to into your MAML document. The link tag will be inserted at the cursor position.

How to insert an external link

External links are links to web pages which are not part of the Brimstone documentation web site. The MAML syntax for creating links is verbose and most easily performed by using code snippet insertion as follows:

  1. Press CTRL+K, CTRL+X to bring up the snippets menu

  2. Press Down Arrow, Enter, Down Arrow, Down Arrow, Enter, then type 'ex' and press Enter again. This will add the link code template at the cursor position.

  3. Type link name and press Tab twice.

  4. Type or paste the link URL.

  5. Delete the <linkAlternateText> tag.

How to insert code

Use the <code> tag to insert blocks of code as follows:

Copy
<code language="cs">
// Any block of code here
</code>

To insert code in the middle of a sentence, use the <codeInline> tag as follows:

Copy
<para>Prefer Linq queries to <codeInline>foreach</codeInline> loops.</para>
<para>Destroy your computer by executing the command: <codeInline>sudo rm -rf /</codeInline>. Note, this is not recommended.</para>
How to insert an image

Images for inclusion in documentation should be added to the media folder, then added to the project in Visual Studio by right-clicking the folder in Solution Explorer and using Add Existing Item... menu option to add it to the project. You can then insert the image into a document as follows:

  1. Open the Entity References window via the View -> Other Windows -> Entity References Window menu option.

  2. Choose Images from the drop-down box to show the list of images in the project.

  3. Drag the node of the image you wish to include into your MAML document. The image code will be inserted at the cursor position.

Documentation style guidelines

New documentation must follow the standards of the existing documentation for consistency. Bear in mind the following points as you write:

  • Place sub-sections inside a <sections> element after the closing </content> tag section of the parent section like this:

    Copy
    <sections>
  <section>
    <title>First sub-section</title>
    <content> ... </content>
  </section>
  <!-- Repeat for as many sub-sections as desired -->
</sections>

  • When using the names of repositories, projects, menu items, window titles, folders and filenames, or emphasizing a word or phrase, surround them in <legacyBold> tags.

  • When introducing new concepts and terminology, surround them in <legacyItalic> tags the first time they are used.

  • When specifying code keywords, variable names or shell commands that are part of another sentence, use the <codeInline> tag to provide a clear separation to the reader between the sentence and the code element.

  • Use ordered lists when listing a sequence of items that are performed in order using <list class="ordered">.

  • Sub-section titles should only use a capital letter for the first character (except for proper pronouns such as Hearthstone).

  • Reference pages and sections that have not been written yet with [TODO: Some topic] so that they are easy to find and create a reference for later in the MAML code.

  • Pages with multiple sections should always include a section outline at the top (you can use the <autoOutline lead="In this section:"/> tag to include this automatically at the end of your introduction. Sub-section outlines are optional.

  • Place tips, notes and warnings immediately under the corresponding paragraph using <alert class="tip/note/warning/caution/important">.

  • All references to Brimstone code elements must be linked so that readers can click the element name to be taken to the relevant API Reference page. To refer to the root API Reference page itself, use <codeEntityReference>R:Project_BrimstoneDocumentation</codeEntityReference>.

  • References to other topics must be linked so that readers can click on the topic name to be taken to the relevant documentation page.

  • Place relevant copies of download links and topic links you used in a page in the <relatedTopics> tag at the end of the page.

Useful tips for documentation contributors

  • The extensions added to Visual Studio by the SHFB installer provide topic document templates, MAML tag auto-completion and tag/attribute suggestions when editing .aml documents, along with code snippets you can insert by pressing CTRL+K, CTRL+X and browsing through the My Xml Snippets folder. Use these to save typing time and discover MAML elements and attributes that may be useful. Pressing the opening chevron (<) will bring up an IntelliSense drop-down showing all of the valid tags for the current position in the document. Finishing an opening tag will automatically insert a closing tag at the cursor. Changing the opening or closing tag's name will change the tag name at the other end of the block automatically.

  • Use CTRL+K, CTRL+F to reformat the current selection or CTRL+K, CTRL+D to reformat the whole document with appropriate indenting. This is very useful when used in combination with the auto-completion above because you do not have to care about making a mess. Insert your code snippets and auto-completions quickly, then reformat the document when you're done.

  • Use the Entity References window to drag tag syntax elements into your document to save typing and working with GUIDs and code element link declarations directly. The window can be used to insert images, topics and API references.

  • Preview your work in the Topic Previewer window before compiling the documentation to save time.

  • Brimstone's solution projects have Sandcastle and MAML code blocks enabled, allowing you to use MAML elements and some proprietary Sandcastle tags such as <inheritDoc/> directly in the code's XML documentation. Use this to make the documentation more readable and maintainable.

  • If you are not sure how to write the tags to render a particular item, try to search through the existing documentation's .aml files to find an already-written example.

  • You can use the standalone Sandcastle Help File Builder project editor to work with documentation if you prefer, however for a smoother and more integrated development experience we strongly recommend using the Visual Studio SHFB extension instead.
