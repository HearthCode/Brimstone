Build Configurations and Flags

[This is preliminary documentation and is subject to change.]

This page summarizes the various build configurations and build flags available in Brimstone.

In this section:

  • Build configurations
  • Build flags
Build configurations

The available build configurations are:

  • Debug - in this configuration, compiler-level code optimizations are disabled, Brimstone's default application-level optimizations are enabled and logging is enabled and will be shown in the console output, or Visual Studio's Debug Output window if your application is started with the debugger.

    To enable or disable specific log items, use the Build flags below. By default, all of the log output build flags are enabled except for _TREE_DEBUG.

    Brimstone runs very slowly in Debug configuration, so should not be used except for testing and debugging purposes. Debug mode is very useful if you wish to follow the flow of a game in a human-friendly readable format.

    Unit tests in BrimstoneTests should not be executed in Debug mode as they will take several hours to complete or longer. Benchmarking with BrimstoneProfiler and Visual Studio Code Analysis profiling should not be performed in Debug mode as the results will be inaccurate.

    Documentation is not built in Debug mode.

  • DebugMono - this is identical to the Debug configuration but excludes projects which are not compatible with Mono, ie. Windows GUI projects like BrimstoneVisualizer which uses WPF. You should use this build configuration when you want a Debug build on Mono.

  • Release - in this configuration, compiler-level code optimizations are enabled, Brimstone's default application-level optimizations are enabled and logging is disabled. Setting log-related build flags will slow down execution but generate no output, therefore you should not use log-related build flags in this configuration.

    Brimstone runs at full speed in Release configuration and is the build mode you should use for finished applications or for speed testing.

    Unit tests in BrimstoneTests should be run in this mode and will complete in under 1 minute. Benchmarking with BrimstoneProfiler and Visual Studio Code Analysis profiling should be performed in this mode for accurate results.

    Documentation is not built in Release mode.

  • ReleaseMono - this is identical to the Release configuration but excludes projects which are not compatible with Mono, ie. Windows GUI projects like BrimstoneVisualizer which uses WPF. You should use this build configuration when you want a Release build on Mono.

  • Documentation - this configuration is based on the Release configuration and all of the notes for Release builds apply, except that a Documentation build will also build the BrimstoneDocumentation project and generate a local copy of the documentation web site in /docs.

    Brimstone takes a long time to build and generates thousands of files in Documentation configuration, so this build mode should only be used for building the documentation web site.

Build flags

The following symbols can be defined when building:

  • _ACTIONS_DEBUG - log standard game actions, eg. playing a card, drawing, attacking, death and so on. Use for debugging the game flow of your scripts

  • _DECK_DEBUG - log pre-game insertion, fill and shuffle operations on a player's deck

  • _GAME_DEBUG - log the creation and resolution of game action blocks, queue underruns, death creation step checks, game step advances and when a game is cloned

  • _QUEUE_DEBUG - log enqueue, dequeue and queue stacking and unwinding actions. Use for debugging the behaviour of the action queue

  • _TREE_DEBUG - log tree search depth, discovery of unique and duplicate game states, search space size and parallelization data

     Note

    This option creates a very large amount of output.

  • _TRIGGER_DEBUG - log game trigger associations, checks, fires, ignores, enqueues and forced runs

  • _ZONE_DEBUG - log all additions and removals of entities to or from a zone
